# AGENTS.md — Live F1 Strategy Co-Pilot

This file is project context for any AI agent (Antigravity/Gemini, or otherwise) working in this repository. Read this in full before doing anything else. If `ROADMAP.md` and this file ever conflict, this file wins on process/discipline; `ROADMAP.md` wins on what's technically in-scope for the current phase.

## What this project is

A multi-agent system that replays real historical F1 races lap-by-lap and generates grounded, on-demand pit-strategy recommendations. It is **not** a live-race tool, a chatbot, or a general F1 commentary system. Full architecture lives in `ROADMAP.md` — read it before writing any code.

Core design principle, non-negotiable: **cheap rule-based math runs field-wide every lap; expensive LLM reasoning only fires on-demand for one selected driver at one selected lap.** Every number that reaches the LLM orchestrator must be traceable to a specific upstream rule-based agent. Never let the LLM invent a statistic, estimate a probability from "general knowledge," or reason about data it wasn't explicitly given in its structured snapshot.

## Process rules (read this before touching code)

1. **Check `ROADMAP.md`'s "Current status" section first.** It states the active phase. Only work within that phase's checklist unless the user explicitly says otherwise.
2. **Do not jump ahead to a later phase's concerns**, even if it seems efficient. If you're implementing Phase 1 (Tire Degradation Engine) and find yourself writing orchestrator prompt code or LLM API calls, stop — that's Phase 6. Flag it as a note instead of building it early.
3. **Do not go backward and silently redesign an earlier, already-checked-off phase.** If you believe an earlier phase's output needs to change, say so explicitly and let the user decide — don't just rewrite it as a side effect of a later task.
4. **Each phase must end in something independently testable and runnable**, per its "Deliverable" line in `ROADMAP.md`. Don't consider a phase done because the code compiles — run it against real FastF1 data and show the output.
5. **When a phase is genuinely complete**, update `ROADMAP.md`: check its boxes, update "Current status" to the next phase. Do this as part of finishing the phase, not as an afterthought.
6. **This is a semester project meant to demonstrate understanding, not just produce working code.** Prefer clear, explainable implementations over clever ones. If there are two reasonable approaches, briefly note the tradeoff rather than silently picking one — the user needs to be able to explain every design decision in a viva/demo.

## Architectural constraints (apply across all phases)

- **No LLM calls in Phases 0-5.** Tire degradation, SC/VSC probability, rival inference, and weather nowcast are pure rule-based math (regression, base-rate statistics, rolling trends) — no exceptions, even for "just a quick heuristic."
- **No tool-calling from the orchestrator.** The LLM synthesis step (Phase 6+) receives a fixed, complete JSON snapshot and nothing else. It does not fetch additional data, call functions, or query anything mid-reasoning. This is a deliberate choice for traceability, not a limitation to work around.
- **Templated, not LLM-generated, `basis` strings.** Short explanatory fields attached to rule-based outputs (e.g. `basis`, `typical_deg_curve_note`) are generated by simple string templates in the rule-based agents themselves — never by calling an LLM to phrase them. This prevents LLM "reasoning" from leaking into what's supposed to be raw grounding data.
- **No raw lap-time arrays reach the orchestrator.** Rule-based agents must pre-digest data into interpreted scores (`degradation_score`, `pit_urgency`, etc.) before it reaches the snapshot. The orchestrator synthesizes over interpreted signals, never does arithmetic itself.
- **Field-wide precompute, single-driver decisions.** All rule-based agents compute for every car, every lap, always — because driver focus is switchable at any point in the demo. Never optimize this away by only computing for a "currently selected" driver.
- **Historical replay only, strictly forward-only.** No future-lap data may leak into a computation for the current lap. If you're unsure whether a piece of FastF1 data is available "at" a given lap in real time, check `data_notes.md` (Phase 0 output) or ask rather than assume.
- **Small-sample honesty.** Any statistic derived from few historical races (e.g. SC rate at a rarely-raced circuit) must be flagged as low-confidence in its output, not presented with false precision.

## Tech stack

- Python, FastF1, pandas, numpy for all rule-based agents and the precompute pipeline
- LLM API (Claude) for the orchestrator only, using structured/JSON-constrained output
- FastAPI, only if the optional Phase 7.5 web backend is built — a thin wrapper around the existing pipeline, explicitly no auth/multi-user/production concerns (see Phase 7.5 in `ROADMAP.md`)
- Local dev on an HP Omen 16 (RTX 4060, 8GB VRAM, 16GB RAM, Windows) — GPU is not required for this project (no local model training or inference; the LLM orchestrator calls an API, not a local model)
- Primary agentic coding environment: Antigravity IDE with Gemini
